<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| PyPI → conda-forge builder | The sdist artifact crosses a supply-chain boundary; conda-forge pins via sha256 |
| conda-forge autotick-bot → feedstock | The bot mutates the feedstock with version+sha256 PRs on every new PyPI release |
| ajstreed maintainer identity → feedstock | Sole maintainer with push; no secondary review on autotick PRs |
| recipe.yaml declarative content → build.script execution | `build.script` runs in the feedstock CI container with build-time privileges |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-13-01 | T (Tampering) | Upstream Cargo dep gets compromised and pulled into the feedstock | mitigate | `cargo-bundle-licenses` audit in Plan 02 enumerates every dep; recipe `source.sha256` pins the exact sdist; Cargo.lock pins exact transitive versions. Every autotick PR re-runs the bundle-licenses step in CI — drift in license fields surfaces immediately. |
| T-13-02 | S (Spoofing) | source.url points at PyPI; attacker could compromise PyPI account and push a malicious sdist | mitigate | PyPI trusted-publisher OIDC (already enabled per quick-task 260415-hb8) prevents credential-theft uploads. sha256 pinning in recipe.yaml detects tampered sdist; conda-forge refuses the build. |
| T-13-03 | R (Repudiation) | Solo ajstreed maintainer — no secondary approver on feedstock PRs | accept | ASVS L1 acceptable for a single-maintainer open-source package. Escalation path documented: if ajstreed is unreachable, the conda-forge core team can add a second maintainer via a PR to the feedstock's `recipe/meta.yaml`/`recipe/recipe.yaml` `extra.recipe-maintainers`. Noted in post-merge verification note. |
| T-13-04 | T (Tampering) | autotick-bot misconfigured post-merge — could open PRs that silently drop files or change the build script | mitigate | Manual watch of the FIRST autotick PR after the initial merge (documented as a follow-up in the post-merge verification note). Autotick diffs are small (version + sha256 + rebuild number); any larger diff is a red flag and must be reviewed line-by-line before merge. |
| T-13-05 | I (Information Disclosure) | THIRDPARTY.yml committed to public repo | accept | Intentional. License text is always public. No secrets or internal data referenced. |
| T-13-06 | E (Elevation of Privilege) | `build.script: python -m pip install . -vv --no-build-isolation` runs maturin which runs cargo which runs proc-macros from arbitrary deps | mitigate | Dep surface already audited in Plan 02. No `build.rs` scripts from unknown authors. Recipe is declarative — no shell injection surface from recipe.yaml authoring. |
| T-13-07 | D (Denial of Service) | A deliberately-malformed sdist (post-compromise) fails cross-compile on a specific platform | mitigate | conda-forge CI builds all target platforms in clean sandboxes per PR; any platform failure blocks merge. The 5-platform matrix acceptance check in Task 4 is the verification gate. |
</threat_model>
