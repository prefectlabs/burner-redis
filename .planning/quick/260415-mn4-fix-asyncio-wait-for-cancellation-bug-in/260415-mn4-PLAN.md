<objective>
Fix asyncio.wait_for cancellation bug in PubSub.get_message

On Python < 3.12, asyncio.wait_for has a known bug (cpython#86296) where task cancellation
can be lost when an external task.cancel() races with the internal timeout machinery. This
affects PubSub.get_message when called with a timeout, which is the common case in
PubSubWorkerThread and any consumer polling loop.

Replace asyncio.wait_for with asyncio.wait, which does not have this race condition.

Purpose: Ensure reliable cancellation behavior across all supported Python versions (3.10+)
Output: Patched pubsub.py with cancellation-safe timeout pattern
</objective>

In PubSub.get_message (line 171), replace the timeout branch (lines 193-197):

```python
                    try:
                        raw = await asyncio.wait_for(self._queue.get(), timeout=timeout)
                    except asyncio.TimeoutError:
                        return None
```

With the cancellation-safe pattern using asyncio.wait:

```python
                    get_task = asyncio.ensure_future(self._queue.get())
                    done, _ = await asyncio.wait({get_task}, timeout=timeout)
                    if done:
                        raw = get_task.result()
                    else:
                        get_task.cancel()
                        try:
                            await get_task
                        except asyncio.CancelledError:
                            pass
                        return None
```

This avoids cpython#86296 where asyncio.wait_for's internal cancellation can swallow an
external task.cancel() on Python < 3.12. asyncio.wait does not cancel the inner task on
timeout -- it simply returns it in the pending set, giving us explicit control over cleanup.

Do NOT change the timeout=None (block forever) or timeout==0.0 (non-blocking) branches.
They are not affected by this bug.
