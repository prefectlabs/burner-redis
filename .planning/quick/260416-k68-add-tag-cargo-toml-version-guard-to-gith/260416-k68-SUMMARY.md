# Quick 260416-k68: Add tag-versus-manifest version guard to release workflow Summary

Added a `verify-version` job to `.github/workflows/release.yml` that compares the pushed `v*` tag against both `Cargo.toml` and `Cargo.lock` versions before any wheels are built, preventing the v0.1.1-style failure where tags were pushed without bumping the manifest.

## Problem

Before this change, `.github/workflows/release.yml` fired on any `v*` tag push and immediately fanned out to four `build` jobs (linux x86_64/aarch64, macOS x86_64/arm64) plus `sdist` and `gate`. If the tag didn't match `Cargo.toml [package].version`, failure only surfaced minutes later at the PyPI publish step (duplicate-version rejection). The v0.1.1 release hit exactly this: tag pushed without bumping `Cargo.toml`, four wheels built against `0.1.0`, PyPI rejected the upload. Recovery required manually bumping the manifest, re-tagging, and re-running the build.

Additionally, partial bumps (editing `Cargo.toml` but not running `cargo build` / `cargo update -p burner-redis`) leave `Cargo.lock` out of sync, breaking reproducibility of the published wheel.

## Solution

One-file edit to `.github/workflows/release.yml` adding a new `verify-version` job at the top of the `jobs:` map with two steps:

**Step 1 — Cargo.toml check:**

```sh
TAG_VERSION="${GITHUB_REF_NAME#v}"
CARGO_VERSION=$(grep -m1 '^version = ' Cargo.toml | sed -E 's/version = "(.*)"/\1/')
# compare, fail with ::error:: and exit 1 on mismatch
```

`grep -m1 '^version = '` anchors to the first `version = ` line, which is always `[package].version` in `Cargo.toml` (other version entries under `[dependencies]` are whitespace-indented and don't match `^version = `).

**Step 2 — Cargo.lock check:**

```sh
LOCK_VERSION=$(grep -A1 '^name = "burner-redis"$' Cargo.lock \
  | grep -m1 '^version = ' \
  | sed -E 's/version = "(.*)"/\1/')
```

`grep -A1 '^name = "burner-redis"$'` locates the unique `burner-redis` package block (only one `[[package]]` declares that name), and `-A1` captures the following `version = ...` line that always follows per Cargo's canonical lockfile format. The outer `grep -m1 '^version = '` strips the `name = ...` line that `-A1` also prints. Fails closed with a "Could not locate" error if upstream Cargo reorders fields within a `[[package]]` block.

Both steps are gated by `if: startsWith(github.ref, 'refs/tags/v')` — defense-in-depth not strictly required since the workflow only fires on `v*` tags, but preserved from the research snippet for safety if the workflow is later extended to other triggers.

**Downstream wiring:**

- `build: { needs: [verify-version], ... }`
- `sdist: { needs: [verify-version], ... }`
- `release: { needs: [verify-version, gate, build, sdist], ... }`

`gate` is NOT gated on `verify-version` — they run in parallel (both have no `needs:`) to minimize wall time. They check orthogonal things: `verify-version` checks manifest-vs-tag, `gate` checks CI passed on the SHA.

## Extraction Commands Chosen

| Target | Extractor | Rationale |
|--------|-----------|-----------|
| `Cargo.toml` | `grep -m1 '^version = ' \| sed -E 's/version = "(.*)"/\1/'` | `^version = ` anchor is unique to `[package].version` (the only non-indented `version =` line). `-m1` stops after first match, making it O(small). |
| `Cargo.lock` | `grep -A1 '^name = "burner-redis"$' \| grep -m1 '^version = ' \| sed -E 's/version = "(.*)"/\1/'` | Block-scoped to the `burner-redis` package via the unique `name = "burner-redis"` line; `-A1` captures the following `version = ...` line that Cargo always writes directly after the name. |

The awk variant (`awk '/^\[\[package\]\]/{pkg=""} /^name = "burner-redis"$/{pkg=1} pkg && /^version = /{...}'`) was rejected in favor of the grep chain: shorter, no awk quoting edge cases, same robustness guarantees.
