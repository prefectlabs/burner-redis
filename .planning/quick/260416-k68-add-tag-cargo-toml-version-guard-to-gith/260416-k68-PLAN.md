From Cargo.toml (line 3):
```
version = "0.1.1"
```
The canonical extractor is:
```sh
grep -m1 '^version = ' Cargo.toml | sed -E 's/version = "(.*)"/\1/'
```
`grep -m1 '^version = '` matches only the first `version = ` line (which is
`[package].version` on line 3 — the second `version = 4` line on Cargo.lock line 3 is
a different file, and inside Cargo.toml any other `version =` lines belong to
`[dependencies]` entries and are prefixed with spaces/quotes, so `^version = ` is
anchored safely to the `[package]` section).

From Cargo.lock (lines 36-40):
```
[[package]]
name = "burner-redis"
version = "0.1.1"
dependencies = [
 "bytes",
```
The lock file has many `name = "..."` and `version = "..."` lines, so extraction
must be block-scoped to the `burner-redis` package. Robust awk extractor:
```sh
awk '/^\[\[package\]\]/{pkg=""} /^name = "burner-redis"$/{pkg=1} pkg && /^version = /{gsub(/[version = "]/, "", $0); print; exit}' Cargo.lock
```
Simpler alternative using grep -A (reads first 3 lines after the matching name line):
```sh
grep -A1 '^name = "burner-redis"$' Cargo.lock | grep -m1 '^version = ' | sed -E 's/version = "(.*)"/\1/'
```
The grep -A1 variant is preferred: shorter, no awk quoting edge cases, and
`^name = "burner-redis"$` is a unique line in Cargo.lock (only one package block
declares that name).

From .github/workflows/release.yml (existing structure):
- Triggered only on `push` tags matching `v*` (line 5-6)
- Four jobs currently: `gate`, `build`, `sdist`, `release`
- `release` already uses `needs: [gate, build, sdist]` (line 110)
- `build` and `sdist` have no `needs:` — they run in parallel with `gate`
- `gate` checks ci.yml + pydocket-compat.yml success on the same SHA via gh CLI
- `release` uses environment: pypi + id-token: write (OIDC trusted publisher)

Since the workflow only fires on `v*` tags, `github.ref` will always match
`refs/tags/v*` — the `if: startsWith(github.ref, 'refs/tags/v')` guard on the step
is defense-in-depth, not strictly required, but keep it to match the research
snippet and to make the step safe if the workflow is later extended to other
triggers.

1. Insert a new `verify-version` job as the first entry under `jobs:` with this
   structure:

   ```yaml
     verify-version:
       runs-on: ubuntu-latest
       steps:
         - uses: actions/checkout@v4

         - name: Verify tag matches Cargo.toml version
           if: startsWith(github.ref, 'refs/tags/v')
           run: |
             TAG_VERSION="${GITHUB_REF_NAME#v}"
             CARGO_VERSION=$(grep -m1 '^version = ' Cargo.toml | sed -E 's/version = "(.*)"/\1/')
             if [ "$TAG_VERSION" != "$CARGO_VERSION" ]; then
               echo "::error::Tag v$TAG_VERSION does not match Cargo.toml [package].version = $CARGO_VERSION"
               exit 1
             fi
             echo "Cargo.toml OK: $CARGO_VERSION"

         - name: Verify tag matches Cargo.lock version
           if: startsWith(github.ref, 'refs/tags/v')
           run: |
             TAG_VERSION="${GITHUB_REF_NAME#v}"
             LOCK_VERSION=$(grep -A1 '^name = "burner-redis"$' Cargo.lock | grep -m1 '^version = ' | sed -E 's/version = "(.*)"/\1/')
             if [ -z "$LOCK_VERSION" ]; then
               echo "::error::Could not locate burner-redis version block in Cargo.lock"
               exit 1
             fi
             if [ "$TAG_VERSION" != "$LOCK_VERSION" ]; then
               echo "::error::Tag v$TAG_VERSION does not match Cargo.lock [[package]] name=\"burner-redis\" version = $LOCK_VERSION (partial bump — did you forget to run cargo update -p burner-redis or cargo build after editing Cargo.toml?)"
               exit 1
             fi
             echo "Cargo.lock OK: $LOCK_VERSION"
   ```

   `jobs:` then `verify-version:` then `gate:` then `build:` then `sdist:` then

Why the two-step guard (Cargo.toml + Cargo.lock):
- Bumping only Cargo.toml without running `cargo build`/`cargo update -p burner-redis`
  leaves Cargo.lock at the old version. This would still pass Cargo.toml check but
  publish a wheel with a mismatched lockfile, breaking reproducibility.
- The Cargo.lock check is cheap (one grep + sed) and the research note explicitly
  calls it out ("Consider also asserting on Cargo.lock to catch partial bumps").

Why `grep -A1 '^name = "burner-redis"$'` works robustly:
- The line `name = "burner-redis"` appears exactly once in Cargo.lock (verified:
  one `[[package]]` block has that name).
- `-A1` grabs the next line, which is always `version = "X.Y.Z"` per Cargo's
  canonical lockfile format.
- The outer `grep -m1 '^version = '` strips the `name = ...` line that `-A1` also
  prints (grep -A prints the match line PLUS the following line; we want only the
  version line).
- If upstream Cargo ever reorders fields within a `[[package]]` block, the guard
  fails closed with the "Could not locate" error rather than silently passing —
  a maintenance-friendly failure mode.
