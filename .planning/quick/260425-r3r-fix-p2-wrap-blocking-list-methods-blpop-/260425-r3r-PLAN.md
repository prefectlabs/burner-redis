<objective>
Fix P2: Make BurnerRedis.blpop / .brpop / .blmove return coroutines (not eager `asyncio.Future`s) so they are compatible with `redis.asyncio.Redis` semantics — usable with `asyncio.create_task(...)` and not started until awaited.

Purpose: PyO3's `pyo3_async_runtimes::tokio::future_into_py(...)` schedules its future onto Tokio at call time and returns a `asyncio.Future`. That breaks two redis.asyncio drop-in expectations:
  1. `asyncio.create_task(r.blpop(...))` raises `TypeError` because `create_task` requires a coroutine.
  2. The blocking pop begins the moment `r.blpop(keys)` is called, not when the result is awaited.

The established codebase fix is to add Python `async def` wrappers in `python/burner_redis/__init__.py` (same pattern already in use for LPUSH / RPUSH / LSET / LREM / LINSERT / SET / SETEX). The `async def` body returns a coroutine and defers the underlying Rust binding call until the coroutine is scheduled.

Output:
  - Three async wrappers in python/burner_redis/__init__.py rebound onto BurnerRedis.blpop, BurnerRedis.brpop, BurnerRedis.blmove.
  - New section comment grouping blocking list wrappers.
  - New tests in tests/test_lists.py covering: create_task acceptance, deferred-execution semantics, and unchanged direct-await results.
  - Existing tests still pass (including test_blpop_cancellation_is_clean — the wrapper makes its inner `_blpop_forever` shim redundant, but the test stays correct).
</objective>

Existing pattern (LPUSH, lines 85-94 of python/burner_redis/__init__.py):

```python
_original_lpush = BurnerRedis.lpush

async def _coerced_lpush(self, name, *values):
    """LPUSH with per-value coercion matching redis-py behavior."""
    coerced = [_coerce_value(v) for v in values]
    return await _original_lpush(self, name, *coerced)

BurnerRedis.lpush = _coerced_lpush
```

The blocking-command wrappers will follow the same shape — capture the original, define `async def`, await it, rebind. They differ from the LPUSH-style wrappers in that they do **no** value coercion (no scalar values are passed to BLPOP/BRPOP/BLMOVE; only keys + timeout + LEFT/RIGHT tokens).

Original signatures from src/lib.rs (must be preserved in wrappers):

```rust
// src/lib.rs:2526-2532
#[pyo3(signature = (keys, timeout=None))]
fn blpop<'py>(&self, py, keys, timeout: Option<f64>) -> ...

// src/lib.rs:2647-2653
#[pyo3(signature = (keys, timeout=None))]
fn brpop<'py>(&self, py, keys, timeout: Option<f64>) -> ...

// src/lib.rs:2761-2769
#[pyo3(signature = (first_list, second_list, timeout, src=None, dest=None))]
fn blmove<'py>(&self, py, first_list, second_list, timeout: f64, src, dest) -> ...
```

Python wrapper signatures (match exactly):

```python
async def _async_blpop(self, keys, timeout=None): ...
async def _async_brpop(self, keys, timeout=None): ...
async def _async_blmove(self, first_list, second_list, timeout, src="LEFT", dest="RIGHT"): ...
```

Note: BLMOVE's Rust signature has `src=None, dest=None` (then internally defaults to "LEFT"/"RIGHT"). The Python wrapper must NOT pre-fill defaults at the wrapper layer — pass through what the caller provided so the Rust default-handling stays the single source of truth. Wrapper signature uses keyword defaults `src="LEFT", dest="RIGHT"` only for redis-py drop-in parity (redis-py's BLMOVE signature uses these defaults). The values are forwarded as-is.
