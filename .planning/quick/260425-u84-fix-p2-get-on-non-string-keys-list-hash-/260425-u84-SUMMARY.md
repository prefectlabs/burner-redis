# Quick Task 260425-u84: Fix P2 GET on non-string keys (list, hash, set, zset, stream) Summary

**One-liner:** `await r.get(non_string_key)` now raises `redis.exceptions.ResponseError("WRONGTYPE Operation against a key holding the wrong kind of value")` instead of silently returning `None`, restoring drop-in compatibility with real Redis for list/hash/set/sorted-set/stream-typed keys.

## What Changed

| Layer | Change |
|-------|--------|
| `src/store.rs` (impl) | `Store::get` signature changed from `Option<Bytes>` to `Result<Option<Bytes>, StoreError>`. Missing/expired keys return `Ok(None)`; live String keys return `Ok(Some(bytes))`; any non-String type returns `Err(StoreError::WrongType)`. Doc comment rewritten to describe the new contract truthfully. |
| `src/store.rs` (StoreError) | Added `PartialEq, Eq` to the existing `derive`. Required by `assert_eq!(store.get(...), Err(StoreError::WrongType))` in unit tests. All variants (`WrongType`, `BusyGroup`, `KeyNotFound`, `NoSuchKey`, plus `NoGroup(String,String)` and `Syntax(String)`) are trivially comparable. |
| `src/store.rs` (unit tests) | Updated 14 `assert_eq!(store.get(&k), Some/None)` assertions to the new `Ok(Some/None)` shape. Rewrote two stale tests (`test_get_on_hash_key_returns_none`, `test_get_on_set_key_returns_none`) and one consistency-renamed test (`test_get_on_sorted_set_returns_none`) to `..._returns_wrongtype` asserting `Err(StoreError::WrongType)`. Added new `test_get_on_list_key_returns_wrongtype` covering the new pain point. Updated section divider comment "GET on non-string types returns None" → "...returns WRONGTYPE". |
| `src/lib.rs` | Both `Store::get` callers (`fn get` pymethod at line 470-482 and `execute_command "get"` arm at line 2936-2943) now call `.map_err(store_err_to_py)?` to propagate WRONGTYPE through the standard `redis.exceptions.ResponseError` Python exception used by every other typed command. |
| `src/persistence.rs` | Updated two `assert_eq!(new_store.get(...), Some(Bytes::from(...)))` round-trip-test assertions to `Ok(Some(...))`. |
| `tests/test_strings.py` | Added `import redis.exceptions`. Appended 5 new regression tests in a `# --- P2-FIX: GET WRONGTYPE on non-string keys ---` section covering list/hash/set/zset/stream. Each uses `pytest.raises(redis.exceptions.ResponseError, match="WRONGTYPE")`. |

## Behavior Change

**User-visible Python API change (intentional drop-in compatibility correction):**

`await r.get(k)` on a key holding a list/hash/set/sorted-set/stream now raises `redis.exceptions.ResponseError("WRONGTYPE Operation against a key holding the wrong kind of value")` instead of returning `None`. This matches real Redis exactly and means application bugs (e.g., a caller that mistakenly `LPUSH`-es to a key it later `GET`-s) surface immediately instead of being silently masked as "no value".

Regression-safe paths (unchanged):
- `await r.get(missing_key)` → `None`
- `await r.get(expired_key)` → `None` (and the entry is passively reaped)
- `await r.get(string_key)` → `bytes`
- `await r.mget(...)` → still returns `None` for non-string members (matches real Redis MGET semantics, intentionally untouched)
- Lua `redis.call('GET', k)` → already raised WRONGTYPE via the independent `src/scripting.rs` dispatch path (untouched)

A search of `tests/` confirmed no existing test relied on the old swallow-None behavior — every `await r.get(...)` callsite in the suite operates on string keys, missing keys, or already-expected-None paths. The only test changes were the two stale Rust unit tests (`test_get_on_hash_key_returns_none` / `..._set_..._none`) called out in the plan, plus the consistency rename of `test_get_on_sorted_set_returns_none`. No downstream Python test required updates.

## Internal API Change

`Store::get` Rust signature changed from `pub fn get(&self, key: &Bytes) -> Option<Bytes>` to `pub fn get(&self, key: &Bytes) -> Result<Option<Bytes>, StoreError>`. This is internal — not part of the public Python API. The two `src/lib.rs` callers and two `src/persistence.rs` test assertions were updated. Sweep-test assertions in `src/store.rs` also follow the new shape (`Ok(Some(...))` / `Ok(None)`).

## Out-of-Scope (Confirmed Untouched)

- **`Store::mget` (`src/store.rs:518-532`)** — Real Redis MGET docs: "If the value stored at any of the keys is not a string … nil is returned." Behavior preserved; no WRONGTYPE for MGET non-string members.
- **`src/scripting.rs` Lua `"GET"` handler (line 355-379)** — Independent command-dispatch path that already raises WRONGTYPE via `RedisValue::Error`. Not modified.
- **`ValueData::String` pattern matches in `src/store.rs` lines 3530, 3633** (persistence-path matches) — Not modified.

## Deviations from Plan

**1. [Rule 3 - Blocking] Added `PartialEq + Eq` derives to `StoreError`**
- **Found during:** First compilation attempt of the rewritten unit tests.
- **Issue:** The plan claimed "`StoreError` derives Debug so this comparison works" — but `assert_eq!` actually requires `PartialEq` (Debug is only used for the failure message). Without it, `assert_eq!(store.get(&k), Err(StoreError::WrongType))` would not compile.
- **Fix:** Added `PartialEq, Eq` to the existing `#[derive(Debug, thiserror::Error)]` line. All `StoreError` variants are unit or `String`, so deriving is trivial and adds no maintenance burden.
- **Files modified:** `src/store.rs` (line 198)
- **Commit:** `acdae17` (single atomic commit with the fix)

**2. [Consistency] Renamed `test_get_on_sorted_set_returns_none` → `..._returns_wrongtype`**
- **Found during:** Updating the sorted-set test's `assert_eq!` shape.
- **Reason:** The plan called out only the hash and set stale tests by name, but the symmetrical sorted-set test had the same naming pattern. Renaming for consistency across all non-string-type GET tests (list, hash, set, sorted-set) makes the suite self-documenting.
- **Files modified:** `src/store.rs`
- **Commit:** `acdae17`

No other deviations — auth gates, architectural changes, or out-of-scope work. The threat model has no new surface.
