<objective>
Fix the silent-`None` bug where `GET` on a non-string-typed key (list/hash/set/zset/stream) returns `None` instead of raising the canonical Redis `WRONGTYPE` error. The fix is purely Rust-side: `Store::get` already detects the wrong-type case but discards the signal — change its signature from `Option<Bytes>` to `Result<Option<Bytes>, StoreError>` and propagate `StoreError::WrongType` through the existing `store_err_to_py` infrastructure that all other commands already use.

Purpose: Real Redis raises `WRONGTYPE Operation against a key holding the wrong kind of value` for `GET` against any non-string type. Burner-redis silently returns `None`, which masks application bugs (e.g. a Prefect caller that mistakenly `LPUSH`-es to a key it later `GET`-s would see "no value" instead of a clear type error). This was a pre-existing bug for hash/set/zset/stream but became user-visible with the recent list support (LPUSH is common in Prefect).

Output: `Store::get` returns `Result`, two lib.rs callers propagate via `store_err_to_py`, two persistence.rs tests updated to the new shape, two stale Rust unit tests in `src/store.rs` rewritten to assert `Err(WrongType)`, and a fresh Python regression test in `tests/test_strings.py` covering all five behavioral paths.
</objective>
