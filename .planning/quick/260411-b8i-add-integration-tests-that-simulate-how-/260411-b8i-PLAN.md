  <action>
Create tests/test_prefect_integration.py with integration tests that simulate how
Prefect's Docket library uses Redis. The tests should be organized into these sections,
each modeling a real Docket workflow:

**1. Sorted Set Queue (Docket's delayed task scheduling)**

Tests that simulate Docket's queue_key sorted set pattern:
- `test_sorted_set_queue_schedule_and_retrieve`: ZADD tasks with timestamps as scores,
  ZRANGEBYSCORE to find due tasks, ZREM to dequeue. This models the scheduler loop
  finding due tasks.
- `test_sorted_set_queue_replace_existing_task`: ZREM old + ZADD new score for the same
  member. This models Docket.replace().

**2. Stream-based Immediate Task Delivery (Docket's stream + consumer group)**

Tests that simulate the worker loop:
- `test_stream_task_delivery_lifecycle`: Full lifecycle: XGROUP CREATE with mkstream,
  XADD task message (fields: key, when, function, args, kwargs, attempt), XREADGROUP to
  claim, XACK to acknowledge. This models Worker._worker_loop main path.
- `test_stream_redelivery_via_xautoclaim`: Add tasks, read with consumer1, then
  XAUTOCLAIM with consumer2 (min_idle_time=0). Verify consumer2 gets the messages.
  This models Worker.get_redeliveries().
- `test_stream_consumer_group_auto_create_on_nogroup`: Create stream via XADD without
  prior XGROUP CREATE, attempt XREADGROUP (expect NOGROUP error), then XGROUP CREATE
  and retry. Models the NOGROUP recovery pattern in get_new_deliveries/get_redeliveries.

**3. Hash-based Execution State (Docket's runs:key hash)**

Tests that simulate execution state management via hashes. Note: `hgetall` and `hexists`
are NOT yet implemented. Mark these tests with `@pytest.mark.xfail(reason="hgetall not
yet implemented")` or `@pytest.mark.xfail(reason="hexists not yet implemented")`.

- `test_execution_state_write_and_read_fields`: HSET runs_key with mapping
  {state, when, known, stream_id, function, args, kwargs}, then HGET each field
  individually. Verifies the hash-based state tracking pattern.
- `test_execution_state_hgetall` (xfail): HSET multiple fields, then HGETALL to read
  all. This models Execution.sync() and Docket.get_execution().
- `test_execution_state_hexists` (xfail): HSET with known field, then HEXISTS to check.
  This models the known_exists check in the schedule Lua script.
- `test_execution_state_transition`: HSET state=scheduled, then HSET state=running
  (overwrite single field), HGET state to verify. Models claim() state transition.

**4. Pipeline Batching (Docket's check_for_work and ack_message)**

Tests that simulate Docket's pipeline usage patterns. Note: `zcard` and `xdel` are NOT
yet implemented. Mark these with appropriate xfail.

- `test_pipeline_check_for_work`: Pipeline with xlen + (attempt zcard -- xfail if
  missing, fall back to manual approach). Models Worker.check_for_work().
- `test_pipeline_ack_message`: Pipeline with xack. Models Worker.ack_message(). Note:
  xdel is not yet available in pipeline, so test the xack-only pattern.
- `test_pipeline_clear_docket`: Pipeline with xtrim(maxlen=0) + delete. Models
  Docket.clear().

**5. Lua Script Atomic Operations (Docket's schedule and cancel scripts)**

Tests that simulate the critical Lua scripts Docket uses:

- `test_lua_schedule_immediate_task`: Lua script that does XADD + HSET atomically.
  Pass KEYS=[stream_key, runs_key] and ARGV with task data. Verify both stream entry
  and hash fields were written. This is a simplified version of Docket's schedule script.
- `test_lua_schedule_delayed_task`: Lua script that does HSET (park data) + ZADD
  (queue) + HSET (runs state) atomically. Models the scheduled (non-immediate) branch.
- `test_lua_cancel_task`: Lua script that does ZREM + DEL + HSET(state=cancelled)
  atomically. Models a simplified Docket._cancel script.
- `test_lua_move_due_tasks_to_stream`: Lua script that does ZRANGEBYSCORE to find due
  tasks, for each: HGETALL the parked data, XADD to stream, DEL parked key, HSET runs
  state=queued. This models the scheduler_loop Lua script. Mark as xfail because
  HGETALL is not yet available in Lua dispatch.

**6. Lock-based Coordination (Docket's per-task-key locking)**

- `test_lock_for_task_scheduling`: Acquire lock on "docket:known:task-key:lock", perform
  schedule operations (HSET + XADD or ZADD), release lock. Verify operations completed
  and lock is released. Models Execution.schedule()'s lock pattern.

**7. Missing Command Identification**

Dedicated xfail tests that clearly document what's missing for full Docket compatibility:

- `test_missing_hgetall` (xfail): Attempt `await r.hgetall("key")` -- expects
  AttributeError. Documents that HGETALL Python API method is needed.
- `test_missing_hexists` (xfail): Attempt `await r.hexists("key", "field")` -- expects
  AttributeError. Documents that HEXISTS Python API method is needed.
- `test_missing_zcard` (xfail): Attempt `await r.zcard("key")` -- expects
  AttributeError. Documents that ZCARD Python API method is needed.
- `test_missing_expire` (xfail): Attempt `await r.expire("key", 60)` -- expects
  AttributeError. Documents that EXPIRE Python API method is needed.
- `test_missing_xdel` (xfail): Attempt `await r.xdel("key", "1-1")` -- expects
  AttributeError. Documents that XDEL Python API method is needed.
- `test_missing_xrange` (xfail): Attempt `await r.xrange("key", "-", "+")` -- expects
  AttributeError. Documents that XRANGE Python API method is needed.

All tests use the `r` fixture from conftest.py. Use `pytest.mark.asyncio` via the
auto mode configured in pyproject.toml. Follow the existing test file patterns
(docstrings on every test, clear section headers with comments).

Each xfail test should use `@pytest.mark.xfail(reason="...", raises=AttributeError)`
with `strict=True` so the test suite alerts when commands are implemented.
