<objective>
Eliminate async overhead in burner-redis command execution by replacing pyo3_async_runtimes::tokio::future_into_py with synchronous execution returning pre-resolved Python awaitables. Add a native Rust pipeline executor to eliminate per-command lock acquisition and Python-Rust boundary crossings.

Purpose: The library is an in-process database with no I/O, but every command currently creates a Tokio future, schedules it, and marshals results through asyncio. This is pure overhead for synchronous operations. Benchmark shows 34.9s vs fakeredis 16.1s for identical workload.

Output: Modified src/lib.rs with ResolvedFuture pattern and execute_pipeline method; updated pipeline.py to use native execution.
</objective>

1. Add a ResolvedFuture pyclass at the top of lib.rs (after the existing helper functions, before the BurnerRedis struct). This is a minimal Python awaitable that returns an already-computed value:

```rust
#[pyclass]
struct ResolvedFuture {
    result: Option<PyObject>,
}

#[pymethods]
impl ResolvedFuture {
    fn __await__(slf: Py<Self>) -> Py<Self> {
        slf
    }
    fn __next__(&mut self, py: Python<'_>) -> PyResult<PyObject> {
        match self.result.take() {
            Some(val) => Err(pyo3::exceptions::PyStopIteration::new_err(val)),
            None => Err(pyo3::exceptions::PyStopIteration::new_err(py.None())),
        }
    }
    fn __iter__(slf: Py<Self>) -> Py<Self> {
        slf
    }
}
```

2. Create a helper function to wrap a PyObject into a ResolvedFuture Bound:

```rust
fn resolved<'py>(py: Python<'py>, value: PyObject) -> PyResult<Bound<'py, PyAny>> {
    Ok(Bound::new(py, ResolvedFuture { result: Some(value) })?.into_any())
}
```

3. Convert ALL commands that currently use `pyo3_async_runtimes::tokio::future_into_py` to the sync fast path pattern EXCEPT:
   - `xreadgroup` when `block` is `Some(...)` -- this uses tokio::select! with stream_notify and tokio::time::sleep
   - `_subscribe_listener` -- this spawns a long-running background Tokio task that blocks on a broadcast channel

For each converted command:
- Remove `let store = self.store.clone();` -- use `&self.store` directly since there's no async move block
- Execute the store operation synchronously inline
- Convert the result to a PyObject
- Return via the `resolved()` helper

Example conversion for GET:
```rust
fn get<'py>(&self, py: Python<'py>, name: &Bound<'py, PyAny>) -> PyResult<Bound<'py, PyAny>> {
    let key = extract_bytes(name)?;
    let result = self.store.get(&key);
    let py_result = match result {
        Some(b) => PyBytes::new(py, &b).into_any().unbind(),
        None => py.None(),
    };
    resolved(py, py_result)
}
```

For commands that return simple types (integers, booleans, strings, Option types), convert directly using `.into_pyobject(py)` or constructing the appropriate Python type.

For commands that build complex Python structures (xpending, xinfo_groups, xinfo_consumers, xautoclaim, xclaim, eval, evalsha, xread, xrange (stream), xreadgroup non-blocking path, etc.) that currently use `Python::try_attach` inside the async block: since we are already holding the GIL (we have `py: Python<'py>`), replace `Python::try_attach(|py| ...)` with direct use of the `py` token. This is simpler and eliminates the `.ok_or_else(|| PyErr...)` unwrap pattern.

For `xreadgroup` specifically: split into two paths:
  - If `block.is_none()`: execute synchronously, return via `resolved()`
  - If `block.is_some()`: keep the existing `future_into_py` path with `store.clone()` and the tokio::select! loop

For `save`: convert to sync -- `store.save()` is file I/O but blocking is acceptable for an explicit save call in an embedded database.

For the `format_xreadgroup_result` helper function: it currently uses `Python::try_attach` because it runs inside an async block without a GIL handle. Create a new version `format_xreadgroup_result_py` that takes a `Python<'py>` parameter for the sync path, and keep the old version for the async blocking xreadgroup path. Or better, refactor `format_xreadgroup_result` to accept an optional `Python` handle. Simplest approach: make a `format_xreadgroup_result_with_py` that takes `py: Python<'_>` directly and builds the same structure, use it from the sync path. Keep the original for the async path.

1. In src/lib.rs, add an `execute_pipeline` method to the BurnerRedis impl block. This method:
   - Accepts a `&Bound<'py, PyList>` of command tuples `(method_name: str, args: tuple, kwargs: dict)`
   - Acquires the store's write lock ONCE (via a new store method or by accessing the internal RwLock directly)
   - Iterates through commands, dispatching each to the appropriate store operation
   - Returns a Python list of results (with errors as exception objects at the appropriate positions, matching redis-py pipeline behavior)

However, acquiring the write lock once is complex because the store methods each acquire their own locks internally. A simpler and still highly effective approach: execute all commands synchronously in a tight loop without the async bridge overhead. Each command still acquires its own lock, but we eliminate:
   - N Tokio future creations
   - N Tokio runtime schedule/wake cycles  
   - N Python asyncio coroutine overhead (await scheduling)
   - N Python-to-Rust-to-Python boundary crossings (just 1 instead)

Implementation:

```rust
fn execute_pipeline<'py>(&self, py: Python<'py>, commands: &Bound<'py, PyList>) -> PyResult<Bound<'py, PyAny>> {
    let results = pyo3::types::PyList::empty(py);
    for item in commands.iter() {
        let tuple = item.downcast::<PyTuple>()?;
        let method_name: String = tuple.get_item(0)?.extract()?;
        let args = tuple.get_item(1)?.downcast::<PyTuple>()?.clone();
        let kwargs = tuple.get_item(2)?.downcast::<PyDict>()?.clone();
        
        // Dispatch to the appropriate command synchronously
        // Each result is either a PyObject or an exception object
        let result = self.dispatch_pipeline_command(py, &method_name, &args, &kwargs);
        match result {
            Ok(val) => results.append(val)?,
            Err(e) => results.append(e.value(py))?,
        }
    }
    resolved(py, results.into_any().unbind())
}
```

Create a `dispatch_pipeline_command` helper method (private, not exposed to Python) that matches on the method name string and calls the store operations directly (not the Python-facing methods, to avoid ResolvedFuture wrapping). This function executes the store operation and converts the result to a PyObject directly.

The dispatch should handle all commands that Pipeline currently supports (check pipeline.py for the full list): set, get, delete, exists, hset, hget, hdel, hvals, hgetall, hexists, hincrby, sadd, smembers, sismember, srem, zadd, zrem, zrange, zrangebyscore, zrangestore, zremrangebyscore, zcard, zscore, zcount, expire, xadd, xread, xlen, xtrim, xdel, xrange, xgroup_create, xgroup_destroy, xreadgroup (non-blocking only), xack, xautoclaim, xclaim, xinfo_groups, xinfo_consumers, xpending_range, eval, evalsha, script_load, script_exists, publish, keys, ttl, setex, mget, xpending.

For any unrecognized command, return a PyException("Unknown pipeline command: {name}") as the error result at that position.

2. In python/burner_redis/pipeline.py, update the `execute()` method to use the native Rust pipeline executor:

```python
async def execute(self):
    """Execute all queued commands and return results in order."""
    if not self._commands:
        return []
    results = await self._client.execute_pipeline(self._commands)
    self._commands = []
    return list(results)
```

The command tuples are already stored as `(method_name, args, kwargs)` which is exactly what the Rust side expects.

Note: The `await` is needed because `execute_pipeline` returns a ResolvedFuture (it goes through the `resolved()` helper). This is still essentially synchronous -- the await resolves immediately on the first `__next__` call.
